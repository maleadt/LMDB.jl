Transactions

Every database operation runs inside a transaction. Transactions may be read-only (MDB_RDONLY) or read-write; an environment supports many concurrent readers but only one writer at a time.

Construction

LMDB.Transaction — Type

A database transaction. Every database operation requires a transaction. Transactions may be read-only or read-write.

A Transaction keeps a reference to its parent Environment, both to expose it via env(txn) and to ensure the env outlives the txn under GC. If a transaction is dropped without an explicit commit or abort, its finalizer aborts it.

source
LMDB.start — Function

Create a transaction for use with the environment

start function creates a new transaction and returns Transaction object. It allows to set transaction flags with flags option.

source

Lifecycle

LMDB.commit — Function

Commit all the operations of a transaction into the database.

The transaction and its cursors must not be used after, because its handle is freed. Idempotent.

source
LMDB.abort — Function

Abandon all the operations of the transaction instead of saving them.

The transaction and its cursors must not be used after, because its handle is freed. Idempotent — safe to call after a previous commit/abort or on a never-opened txn.

Read-only reuse

For read-only transactions, the txn handle can be parked across requests to skip the begin/abort cost:

Base.reset — Method

Reset a read-only transaction

Abort the transaction like abort, but keep the transaction handle.

source
LMDB.renew — Method

Renew a read-only transaction

This acquires a new reader lock for a transaction handle that had been released by reset. It must be called before a reset transaction may be used again.

source

Sub-transactions

Pass parent = txn to start to nest a child write transaction inside an open write transaction. The child sees the parent's uncommitted state; on commit the child's changes are folded into the parent, on abort they are discarded.
